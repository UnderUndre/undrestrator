# Specification Quality Checklist: Internationalization & Localization

**Purpose**: Validate that the feature is built for multi-locale from day one — not retrofitted after a sales call.
**When to use**: Any product targeting users outside a single English-speaking market. Any feature touching strings, dates, numbers, currency, names, addresses, content layout.
**When to skip**: Internal English-only tools with no localization roadmap. Even then — date formatting and timezone handling apply.

## Validation Items

### Translation Pipeline

- [ ] CHK001 - Is the translation-source strategy specified (extracted strings file, in-code with i18n hook, CMS-managed)? [Clarity]
- [ ] CHK002 - Are requirements explicit that NO user-facing string is hardcoded in components? [Completeness]
- [ ] CHK003 - Is the translation-key naming convention specified (namespaced, hierarchical, snake_case vs camelCase)? [Consistency]
- [ ] CHK004 - Is the missing-translation fallback policy specified (fall through to default locale, show key, throw)? [Edge Case]
- [ ] CHK005 - Is the translation-update workflow documented (who edits, where, how it lands in production)? [Gap]

### Locale Detection & Persistence

- [ ] CHK006 - Is locale-detection priority specified (URL > user pref > `Accept-Language` > geo-IP > default)? [Clarity]
- [ ] CHK007 - Is locale persistence specified (cookie, server-side user pref, URL prefix)? [Completeness]
- [ ] CHK008 - Are URL-locale conventions specified (subpath `/de/`, subdomain `de.`, query, none)? [Clarity]

### Pluralization & Grammar

- [ ] CHK009 - Are pluralization requirements specified using ICU MessageFormat (or CLDR-aware mechanism)? [Completeness]
- [ ] CHK010 - Are gender / grammatical-case requirements explicit for languages that need them (Slavic, Semitic)? [Coverage]
- [ ] CHK011 - Are interpolation requirements specified to avoid string concatenation across languages? [Clarity]

### Formatters

- [ ] CHK012 - Are date / time formatting requirements specified using `Intl.DateTimeFormat` (no hardcoded patterns)? [Completeness]
- [ ] CHK013 - Is timezone handling specified per data point (UTC store, locale display, user pref override)? [Coverage]
- [ ] CHK014 - Are number-formatting requirements specified (locale-aware grouping, decimal separators)? [Clarity]
- [ ] CHK015 - Are currency requirements specified (per-locale, fallback for unsupported, decimal precision)? [Completeness]
- [ ] CHK016 - Is the relative-time formatting requirement specified (`Intl.RelativeTimeFormat`)? [Completeness]

### Right-to-Left (RTL)

- [ ] CHK017 - Are RTL languages explicitly in scope or out (Arabic, Hebrew, Persian, Urdu)? [Clarity]
- [ ] CHK018 - If RTL in scope, are layout requirements specified (logical CSS properties: `margin-inline`, `padding-inline`)? [Completeness]
- [ ] CHK019 - Are RTL component-mirroring requirements specified (icons that must mirror vs not)? [Edge Case]
- [ ] CHK020 - Is bidirectional-text handling specified (mixed LTR/RTL content, embedded numbers in RTL)? [Coverage]

### Content & Length

- [ ] CHK021 - Are layout requirements explicit that translated strings can be 30-200% longer (German, Russian) — no fixed-width truncation? [Edge Case]
- [ ] CHK022 - Are line-break / hyphenation requirements specified for languages without word spacing (CJK, Thai)? [Gap]
- [ ] CHK023 - Are font-stack requirements specified per language (CJK fonts, system fonts vs web fonts)? [Completeness]

### Cultural & Regional

- [ ] CHK024 - Are name-format requirements specified (single field for all locales vs separate first/last)? [Clarity]
- [ ] CHK025 - Are address-format requirements specified per region (or generic free-form)? [Coverage]
- [ ] CHK026 - Are calendar-system requirements specified (Gregorian only, or also Hijri / Hebrew / Buddhist)? [Gap]

### SEO & URLs

- [ ] CHK027 - Are `hreflang` requirements specified for multi-locale pages? [Completeness]
- [ ] CHK028 - Are URL-slug requirements specified (transliterated vs translated, per-locale slug catalog)? [Clarity]

### Testing & QA

- [ ] CHK029 - Is the pseudo-localization strategy specified for catching layout overflow before real translations land? [Gap]
- [ ] CHK030 - Are RTL visual-regression tests required if RTL is in scope? [Coverage]
- [ ] CHK031 - Is the translation-coverage gate specified (% strings translated before release, fallback for the rest)? [Measurability]

## Notes

- Reference: CLDR (Unicode locale data), ICU MessageFormat, ECMA-402 (`Intl` namespace).
- "i18n" = code is translation-ready. "L10n" = the translations exist. Both have separate work and separate gates.
